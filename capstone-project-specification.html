1. Business Use Case — Why Agentic AI?

1.1 The Company

Prodapt is a mid-size telecom provider offering mobile voice, 4G LTE, and 5G data services across the United States. It serves consumer, business, and enterprise customers. Like every telecom operator, Prodapt receives thousands of inquiries daily about network problems, billing disputes, roaming, device upgrades, and service-level agreements (SLAs).

Today, these inquiries are handled by separate teams using separate tools. Your capstone project automates this fragmented workflow using a multi-agent AI Operations Center.

1.2 Who Uses the System?

Customer Support Agent (primary user)
Front-line staff who receive customer calls and chats. They need fast, accurate answers and often must coordinate between policy documents, network data, and billing systems — usually while the customer waits on hold.
Network Operations (NOC) staff
Engineers who monitor tower status, diagnose connectivity issues, and manage open incidents. Their expertise is technical; customer-facing explanations require a different tone.
Billing Resolution team
Specialists who investigate duplicate charges, apply credits, and follow dispute policy. Actions must comply with credit limits and leave an audit trail in the billing database.
End customer (indirect beneficiary)
Subscribers who expect one answer — not three transfers — when they ask: "My 5G keeps dropping and I was charged twice — am I owed a credit?"

1.3 The Existing Problem (As-Is — Manual Operations)

Before automation, a single customer inquiry can touch four disconnected systems and three different teams. There is no intelligent routing — humans decide where to look.

Today's pain points

  • Fragmented knowledge — Policies live in PDF/TXT documents; outage and tower data live in SQL databases; live diagnostics sit in NOC dashboards; billing data sits in a separate CRM/billing system.
  • Slow resolution — An agent puts the customer on hold, searches policy folders, emails NOC, waits for a billing supervisor, then manually drafts a reply. Average handle time: 15–25 minutes for complex cases.
  • Inconsistent answers — Two agents may interpret the roaming policy or SLA credit rules differently because answers are not grounded in the same source documents.
  • Wrong-team routing — Billing agents cannot diagnose tower issues; NOC engineers should not apply credits. Customers get transferred multiple times.
  • Multi-domain questions fail — Questions like "We had a 6-hour Midwest outage — am I eligible for SLA credit?" require both operational data and policy interpretation. No single tool handles both.
  • High training cost — New hires need weeks to learn which system to use for which question type.
  • No audit trail — It is hard to prove which data sources and policies were consulted when a credit was applied or a diagnosis was given.

What the business needs

  • One entry point where staff (or customers) ask a question in plain English
  • Automatic routing to the right specialist — policy, analytics, network, or billing
  • Answers grounded in actual policy documents and live database records
  • Actionable outcomes — diagnostics run, credits applied — with data persisted in SQL
  • Professional, consistent customer-ready responses every time
  • Visibility into which agents ran and what they found (execution trace)
  • Ability to scale specialist services independently (network vs billing microservices)

1.4 The Expected Solution (To-Be — Agentic AI Operations Center)

You will build an AI Operations Center that acts as an intelligent front door. A LangGraph supervisor understands the intent of each inquiry and delegates to specialist workers — each powered by the right framework for that job. The result is a single, coherent, customer-ready response.

6. Detailed Build Instructions (No Code — Design Only)

6.1 LlamaIndex Document RAG (document_rag.py)

  1. Configure a local embedding model (HuggingFace all-MiniLM-L6-v2 is recommended from your lab).
  2. Load all TXT files from data/documents/ using SimpleDirectoryReader.
  3. Build a VectorStoreIndex and persist embeddings to data/vector_index/ so rebuilds are not required every run.
  4. Expose a single function that accepts a natural language question and returns a string answer.
  5. Use as_query_engine — not LlamaIndex agents, workflows, or ReAct patterns.

6.2 LlamaIndex Semantic SQL (sql_semantic_search.py)

  1. Connect to data/telecom_ops.db using the LlamaIndex SQLDatabase wrapper.
  2. Create SQLTableNodeMapping for all tables in the schema.
  3. Define SQLTableSchema objects with context strings for semantic table selection.
  4. Build ObjectIndex over table schemas using VectorStoreIndex.
  5. Create SQLTableRetrieverQueryEngine with similarity_top_k of 1 or 2.
  6. Expose a single function that accepts a question and returns the synthesized answer.

6.3 Network Diagnostics ADK Service (port 8001)

  1. Create a Google ADK Agent with a clear instruction describing NOC diagnostics responsibilities.
  2. Implement three async tools — each tool opens a SQLite connection and runs SQL:
    • check_tower_status(tower_id) — returns status, technology, signal metrics, any open incident
    • run_connectivity_diagnostics(tower_id, symptom) — reads tower data from SQL, returns recommendations
    • get_regional_network_summary(region) — aggregates tower health for a region from SQL
  3. Wrap the agent with to_a2a and run with uvicorn on port 8001.
  4. Verify the agent card is published at localhost:8001/.well-known/agent-card.json

6.4 Billing Resolution ADK Service (port 8002)

  1. Create a Google ADK Agent for billing dispute resolution.
  2. Implement three async tools — all SQL-backed:
    • lookup_billing_account(customer_id) — balance and recent charges from SQL
    • check_duplicate_charges(customer_id) — find flagged duplicates in billing_charges
    • apply_billing_credit(customer_id, amount, reason) — INSERT credit row, UPDATE account balance; use PENDING_APPROVAL in SQL when amount exceeds policy auto-approval limit ($50 per billing_disputes_policy.txt)
  3. Expose via to_a2a on port 8002.

6.5 LangGraph Remote ADK Client (adk_remote_client.py)

  1. Create RemoteA2aAgent instances pointing to each agent card URL.
  2. Wrap each remote agent in a lightweight proxy ADK Agent with the remote as sub_agent.
  3. Use ADK Runner.run_async to send messages and collect final responses.
  4. Expose synchronous wrapper functions for use inside LangGraph nodes (Network Diagnostics and Billing Resolution).
  5. Handle service-unavailable errors gracefully with a clear message if ports 8001/8002 are not running.

6.6 CrewAI Customer Communications (crew_nodes.py)

  1. Create a two-agent sequential crew:
    • Communications Specialist — drafts response using original query + accumulated agent_context from prior workers
    • Quality Reviewer — reviews for accuracy, tone, and policy compliance; outputs final text only
  2. Use Process.sequential. Reviewer task depends on drafter task output.
  3. Expose one function: accepts user query and context string, returns final customer-ready response.

6.7 LangGraph Orchestration (graph.py + state.py)

State (AgentState) must include:

  • messages — conversation history (append-only across nodes)
  • next — supervisor routing decision
  • user_query — original question
  • agent_context — accumulated outputs from all workers (passed to CrewAI)

Worker nodes to implement:

Node NameActionReturns To

6.8.3 Main area — inquiry, response & Agent Execution Trace

Required — Agent Execution Trace

This is a mandatory demo and evaluation feature. Evaluators use it to verify correct routing (PolicyRAG vs NetworkAnalytics vs ADK workers vs CustomerCommsCrew) without reading server logs.

  1. Customer inquiry input — multi-line text area; placeholder text inviting plain-English questions; bound to session state if you use programmatic updates elsewhere
  2. Submit button — invokes your graph entry function (e.g. run_telecom_assistant(user_query)); show a spinner while the graph runs
  3. Final customer response — display final_response prominently (e.g. success/info box) after the run completes; this is the CrewAI-polished text shown to the customer
  4. Agent Execution Trace — an expander (recommended: expanded by default on results) that lists every worker executed in order. For each step show:
    • Step number (1, 2, 3, …)
    • Worker name exactly as in LangGraph (PolicyRAG, NetworkAnalytics, NetworkDiagnosticsADK, BillingResolutionADK, CustomerCommsCrew)
    • Truncated worker output — the text that worker contributed to agent_context (truncate long outputs for readability, e.g. first 500 characters)
  5. Empty trace handling — if no workers ran, show a clear message inside the trace panel
  6. Error handling — if the graph throws, show st.error with the message; do not leave the UI in an ambiguous state
